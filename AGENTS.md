## 1. 规范来源

本项目执行 `LLM-Rule/AGENTS.md` v3 中的 AI 编程规范。核心要求：

- 目标明确、范围明确、可验证、文档同步。
- 所有项目文档、配置示例和运行路径示例使用相对路径。
- 端口必须可配置，应用服务端口不使用常用端口。
- Docker 部署必须可用，至少支持 Linux OS 部署运行。
- 功能、接口、配置、数据库、部署、页面交互或测试方式变更后，必须同步更新对应文档。
- 开发过程使用 Todo 管理，并维护 `docs/ProcessTodo.md`。
- 修改代码后必须运行必要测试或检查；无法运行时说明原因、影响和替代验证方式。

## 2. 项目事实

- 项目名称：ItemMark。
- 项目定位：统一的物品标签打印与扫码查询系统。
- Web 框架：FastAPI + Jinja2 模板。
- 同步服务：独立 `sync` 容器执行模块同步。
- 应用端口：`WEB_PORT=18089`，`HOST_WEB_PORT=18089`。
- 默认域名：`https://code.zestrade.com`。
- 首版模块：`material-info`、`wuping`。
- 旧二维码兼容路径：`/material/{code}` 归属 `material-info` 模块。
- 武平扫码路径：`/wuping/material/{code}`。

## 3. 本项目交付规则

每次任务开始前必须明确：

- 本次目标。
- 成功标准。
- 允许修改范围。
- 明确不做范围。
- 影响的页面、接口、数据、配置、部署或文档。
- 验证方式。

任务粒度应保持小步闭环。发现无关问题时，优先记录到 `docs/ProcessTodo.md` 或在交付说明中反馈，不擅自扩大修改范围。

## 4. 文档维护规则

权威文档：

| 文档 | 用途 |
|---|---|
| `README.md` | 项目入口、运行方式、常用入口、API 和部署摘要 |
| `PRD.md` | 产品需求入口，指向当前 PRD 主文档 |
| `docs/01-项目整合PRD.md` | 当前产品需求主文档 |
| `docs/02-程序设计实现方案.md` | 当前技术设计主文档 |
| `docs/Index.md` | 文档索引和维护规则 |
| `docs/ProcessTodo.md` | 任务 Todo、阻塞项、待用户提供内容 |
| `docs/Config.md` | 配置项说明 |
| `docs/Deployment.md` | 部署说明 |
| `docs/DatabaseSchema.md` | 数据库说明 |
| `docs/TestReport.md` | 测试记录和遗留风险 |

文档规则：

- 文档必须有目录。
- 同一事实只能有一个主文档作为权威来源，其他文档摘要或引用。
- 暂未确认的内容标记“待确认”。
- 废弃内容标记“已废弃”，并说明替代文档或废弃原因。
- 不在文档中写真实密钥、真实 Token、真实密码或生产隐私数据。

## 5. 开发与验证规则

常用验证命令：

```bash
python3 -m py_compile sync/*.py web/*.py web/routers/*.py web/services/*.py web/db/*.py
docker-compose config
```

本地启动：

```bash
bash scripts/start-dev.sh
```

本地停止：

```bash
bash scripts/stop-dev.sh
```

健康检查：

```bash
curl http://127.0.0.1:18089/api/health
```
